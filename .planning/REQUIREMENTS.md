# Requirements: 해군 사상자 관리 전산 시스템

**Defined:** 2026-03-31
**Core Value:** 허가된 담당자만 접근 가능한 정확한 기록 — 군 인사기록 특성상 한 번 등록된 사상자·상이자 기록은 영구 보존되고 감사 추적이 가능해야 한다.

---

## v1 Requirements

### 인증 및 접근 제어 (AUTH)

- [x] **AUTH-01**: 담당자가 사용자 ID/비밀번호로 로그인할 수 있다
- [x] **AUTH-02**: 로그인 실패 5회 시 계정이 잠금되고, 관리자만 해제할 수 있다
- [x] **AUTH-03**: 30분 무조작 시 세션이 자동 만료된다
- [x] **AUTH-04**: 동시 세션을 1개로 제한하여 중복 로그인을 방지한다
- [x] **AUTH-05**: 관리자가 역할(ADMIN/MANAGER/OPERATOR/VIEWER)을 사용자에게 부여할 수 있다
- [x] **AUTH-06**: 역할에 따라 메뉴·기능 접근이 제한된다 (RBAC)
- [x] **AUTH-07**: 강제 로그아웃(세션 즉시 무효화)이 가능하다

### 감사 및 보안 (AUDIT)

- [x] **AUDIT-01**: 모든 조회·등록·수정·삭제·출력·Excel 다운로드 행위가 자동으로 감사 로그에 기록된다 (누가, 언제, 어떤 레코드, 어떤 작업, IP 주소)
- [x] **AUDIT-02**: 감사 로그는 일반 사용자가 수정·삭제할 수 없다 (append-only)
- [x] **AUDIT-03**: 감사 로그는 최소 5년 이상 보관된다 (해외 DoD 기준 적용)
- [x] **AUDIT-04**: 주민등록번호는 AES-256 암호화로 저장된다 (개인정보보호법 제24조의2 의무)
- [x] **AUDIT-05**: 화면에서 주민등록번호는 역할에 따라 마스킹 표시된다 (VIEWER: 전체 마스킹, OPERATOR: 뒷자리 마스킹, MANAGER+: 전체 노출)
- [x] **AUDIT-06**: 모든 삭제는 논리 삭제(soft delete)로 처리되고, 삭제 사유를 필수 입력받는다 (공공기록물관리법 영구 보존 의무)
- [x] **AUDIT-07**: 수정 이력이 별도 이력 테이블에 보관된다 (공공기록물관리법 제20조)

### 사망자 관리 (DEAD)

- [x] **DEAD-01**: 담당자가 사망자 목록을 조회할 수 있다 (군구분·군번·성명·생년월일·계급·소속 복합 검색)
- [x] **DEAD-02**: 담당자가 사망자 정보를 등록할 수 있다 (군번, 성명, 주민등록번호, 계급, 소속, 입대일자, 전화번호, 사망구분, 사망구분 기호, 주소, 사망일자)
- [x] **DEAD-03**: 담당자가 사망자 정보를 수정할 수 있다
- [x] **DEAD-04**: 관리자가 사망자 정보를 논리 삭제할 수 있다 (삭제 사유 필수)
- [x] **DEAD-05**: 사망자 현황을 Excel 파일로 다운로드할 수 있다 (검색 조건 적용, 주민번호 마스킹)
- [x] **DEAD-06**: 사망 기록에 상태(Status) 필드를 관리한다 (등록→확정→보훈청통보완료)
- [x] **DEAD-07**: 중복 등록을 방지한다 (군번 또는 주민번호 기준)

### 상이자 관리 (WOND)

- [x] **WOND-01**: 담당자가 상이자 목록을 조회할 수 있다 (군구분·군번·성명·생년월일·계급·소속 복합 검색)
- [x] **WOND-02**: 담당자가 상이자 정보를 등록할 수 있다 (군번, 성명, 주민등록번호, 계급, 소속, 입대일자, 전화번호, 현주소, 보훈청명, 병명, 상이구분[전공상/공상/일반상이])
- [x] **WOND-03**: 담당자가 상이자 정보를 수정할 수 있다
- [x] **WOND-04**: 관리자가 상이자 정보를 논리 삭제할 수 있다 (삭제 사유 필수)
- [x] **WOND-05**: 상이자 현황을 Excel 파일로 다운로드할 수 있다
- [x] **WOND-06**: 상이 기록에 상태(Status) 필드를 관리한다 (등록→심사중→확정→보훈청통보완료)
- [x] **WOND-07**: 보훈청명은 코드 테이블에서 선택한다 (전국 보훈청/보훈지청 목록)

### 전공사상심사 관리 (REVW)

- [x] **REVW-01**: 담당자가 전공사상심사 현황을 조회할 수 있다 (군구분·군번·성명·생년월일·계급·소속 복합 검색)
- [x] **REVW-02**: 담당자가 전공사상심사 정보를 등록할 수 있다 (심사차수, 심사일자, 성명, 군번, 주민번호, 입대일자, 소속, 병명, 소속부대 심사결과, 전공상 분류[전공상/공상/기각/보류])
- [x] **REVW-03**: 담당자가 전공사상심사 정보를 수정할 수 있다
- [x] **REVW-04**: 관리자가 전공사상심사 정보를 논리 삭제할 수 있다 (삭제 사유 필수)
- [x] **REVW-05**: 전공사상심사 현황을 Excel 파일로 다운로드할 수 있다
- [x] **REVW-06**: 심사차수별 이력을 전체 조회할 수 있다 (심사 이력 Chain 보존)
- [x] **REVW-07**: 심사 결과 변경 시 사망자/상이자 분류에 자동 반영된다
- [x] **REVW-08**: 보훈청 통보 일시를 기록할 수 있다 (군인재해보상법 제44조)

### 공식 문서 출력 (DOCU)

- [x] **DOCU-01**: 국가유공자 요건 해당사실 확인서(사망자용)를 출력할 수 있다 (현행 해군 공식 양식)
- [x] **DOCU-02**: 국가유공자 요건 해당사실 확인서(상이자용)를 출력할 수 있다 (현행 해군 공식 양식)
- [x] **DOCU-03**: 전공사상심사결과서를 출력할 수 있다
- [x] **DOCU-04**: 순직/사망확인서를 출력할 수 있다
- [x] **DOCU-05**: 사망자 현황 보고서를 출력할 수 있다
- [x] **DOCU-06**: 상이자 현황 보고서를 출력할 수 있다
- [x] **DOCU-07**: 전사망자 확인증 발급대장을 출력할 수 있다
- [x] **DOCU-08**: 문서 출력 전 발급 목적을 필수 입력받고, 발급 이력을 자동 기록한다 (국가유공자예우법 제80조)
- [x] **DOCU-09**: 인쇄 미리보기를 제공한다

### 통계 및 현황 (STAT)

- [x] **STAT-01**: 신분별 사망자 현황을 조회하고 Excel로 저장할 수 있다
- [x] **STAT-02**: 월별 사망자 현황을 조회하고 Excel로 저장할 수 있다
- [x] **STAT-03**: 연도별 사망자 현황을 조회하고 Excel로 저장할 수 있다
- [x] **STAT-04**: 부대별 사망자 현황을 조회하고 Excel로 저장할 수 있다
- [x] **STAT-05**: 부대별 사망자 명부를 조회하고 Excel로 저장할 수 있다
- [x] **STAT-06**: 전사망자 명부를 조회하고 Excel로 저장할 수 있다
- [x] **STAT-07**: 통계 조회 시 응답시간이 5초 이내여야 한다

### 시스템 기반 (INFRA)

- [x] **INFRA-01**: Docker Compose로 단일 명령 배포가 가능하다 (군 내부망 에어갭 환경)
- [x] **INFRA-02**: PostgreSQL 16 기반 DB 스키마가 Flyway 마이그레이션으로 관리된다
- [x] **INFRA-03**: spring-session-jdbc 기반 세션 저장소를 사용한다 (JWT 미사용)
- [ ] **INFRA-04**: 모든 외부 라이브러리 의존성이 오프라인에서 해결 가능하다 (Nexus/로컬 Maven 레포)
- [x] **INFRA-05**: HTTPS(TLS) 전송 암호화가 적용된다

## v2 Requirements

### 첨부파일 관리
- **ATTC-01**: 사망·상이·심사 기록에 관련 문서(의무기록, 목격자 진술 등) 첨부 가능
- **ATTC-02**: 첨부파일 버전 관리 (VBMS 사례 참고)

### 워크플로우 고도화
- **FLOW-01**: 상이자 심사 결과 이의신청 이력 관리
- **FLOW-02**: 승인(Approve) 단계 도입 — 등록 후 상급자 승인 워크플로우

### 공식 문서 전자서명
- **ESIG-01**: PKI 기반 전자서명 적용 (영국 JPA 사례 참고)

### 통계 고도화
- **STAT-08**: 읽기 전용 DB replica 연결로 통계 쿼리 성능 분리
- **STAT-09**: 배치 집계 테이블 — 야간 집계로 대시보드 성능 개선

### 개인정보 고도화
- **PII-01**: 주민번호 별도 클릭 시만 노출 (DoD DoDI 5400.11 Category 1 기준)
- **PII-02**: 암호화 키 로테이션 — 연간 키 교체 체계

## Out of Scope

| Feature | Reason |
|---------|--------|
| 보훈청 API 실시간 연동 | 군 내부망과 보훈청 시스템 간 네트워크 분리. 별도 사업 영역 |
| 급여·보상금 산출 | 보훈청 업무 영역. 이 시스템은 사실 확인만 담당 |
| 모바일 앱 | 군 내부망 데스크톱 전용 환경 |
| 외부 인터넷 연동 | 에어갭 군 내부망 배포 원칙 |
| 타군(육·공·해병) 데이터 관리 | 해군 단독 배포. DIMHRS 실패 교훈 반영 — 범위 과팽창 방지 |
| 실시간 알림/Push | 조회·등록 시스템에 불필요. 복잡도 대비 가치 낮음 |
| 클라우드 SaaS 구성 | 군 내부망 온프레미스 전용 |
